# Admin API Contract

## Failure-Aware Upstreams

Proxy-host create/update accepts additive `retry`, `passive_health`, and
per-upstream `administrative_state`. Omission means retry disabled, passive health
disabled, and active upstream. Session, CSRF, validation, revision apply,
acknowledgement, rollback and audit remain mandatory; no direct-memory drain
endpoint exists. Authenticated `/api/v1/upstream-health` returns effective health,
nullable drain state (`active`, `draining`, `drained`, `removed`) and connection
count without endpoint or connection identity.

Base path: `/api/v1`

For a command-line operator flow that does not use the Admin Web UI, see
`docs/admin-curl.md`.

## Current Status

The `edge-admin-api` crate currently defines the API contract, DTOs, auth/session
models, CSRF guard, error schema, and `ConfigLifecycle`/`CoreCommandClient`
boundary tests. It also has a framework-free HTTP request/response contract
router for status and mutation auth preflight, plus raw HTTP parse/render
helpers for the bin-bound listener. `apps/edge-proxy` binds the Admin
HTTP listener during config-file startup for status, login/logout, and mutation
auth preflight. Login emits a secure/httpOnly/sameSite session cookie and CSRF
token. Setup writes the first admin password hash through `SecretStore` when no
hash exists. `POST /api/v1/proxy-hosts` has a framework-free HTTP contract
handler that verifies session/CSRF, parses the JSON DTO, validates the next
config, and calls `ConfigLifecycle::apply_with_core`. The bin-bound TCP listener
now binds this create endpoint to file revisions and the `run_snapshot_http_proxy_mio`
runtime command channel. The running data plane applies `ApplyConfigSnapshot`
before returning accepted acknowledgement, and the lifecycle commits `current`
only after that acknowledgement. `POST /api/v1/config/rollback` is also bound
through the same lifecycle/runtime command boundary. `GET /api/v1/proxy-hosts`,
`GET /api/v1/proxy-hosts/{id}`, `PATCH /api/v1/proxy-hosts/{id}`, and
`DELETE /api/v1/proxy-hosts/{id}` now read/update/remove generated route/service
through the same API contract boundary. `GET /api/v1/config` returns the current
rendered MVP config and `POST /api/v1/config/validate` validates a raw TOML MVP
config body without applying it. `POST /api/v1/config/diff` returns a validated
diff for a raw TOML candidate, and `POST /api/v1/config/apply` applies a raw
TOML candidate through the lifecycle/runtime command boundary. `GET /api/v1/health`
returns a minimal operational health response without secrets or upstream URLs.
Authenticated `GET /api/v1/upstream-health` returns the active health generation
and ordered per-upstream availability without endpoint URLs or probe details.
`GET /api/v1/certificates`, `GET /api/v1/certificates/{id}`,
`GET /api/v1/logs/access`, and `GET /api/v1/logs/errors` are bound as
session-protected read-only observability endpoints. Data-plane access logs are
fed into the Admin recent access buffer through a bounded nonblocking queue, and
data-plane 502/504 errors are fed into the Admin recent error buffer through the
same boundary. Bound Admin mutation paths record `RUNTIME_COMMAND_REJECTED`
responses into the Admin recent error buffer. Runtime log queue-full drops are
exposed as `RUNTIME_LOG_QUEUE_FULL` recent errors only in `field-debug` and
`dev` modes. Runtime metrics are generated by the snapshot mio runtime, health
runtime, and bootstrap producers and handed through a bounded nonblocking queue
to a single-writer registry. Authenticated `GET /api/v1/metrics` returns a
bounded immutable summary; the optional loopback Prometheus listener exposes
the same snapshot independently at exact `GET /metrics`.

Authenticated `GET /api/v1/audit` queries the durable reader port. It defaults
to 50 records, caps at 100, accepts only exact typed `action`, `outcome`,
`target_kind`, `from`, `to`, `limit`, and opaque `cursor` parameters, and never
returns frame hashes, filesystem paths, raw config, payload bodies, or secrets.
`POST
/api/v1/certificates/{id}/issue` and `POST /api/v1/certificates/{id}/renew`
are also bound. They call the application certificate issue/renew use cases
through `AcmeClient`, `CertificateStore`, and `AuditSink`, then submit
`InstallCertificate` through `CoreCommandClient` before returning success. In
the config-file startup path, issue receives an HTTP-01 token/key authorization
from the selected ACME adapter, registers it in the shared challenge store,
verifies that token through the runtime HTTP listener, and clears it on success
or failure. The default automatic-gate adapter is fake and returns
`fake-acme-staging` or `fake-acme-production` as `source`; these responses must
not be used as external Let's Encrypt staging evidence. Starting the process
with `SPONZEY_ACME_CLIENT=letsencrypt-staging` wires the real staging adapter
for deferred Post-MVP work, which returns `letsencrypt_staging` on success.
`edge-proxy` injects a
file-backed `CertificateStore` rooted at `data/certs`, so issued/renewed
certificates are persisted as `fullchain.pem`, `privkey.pem`, and
`metadata.toml` under the target certificate ref.
The `edge-proxy` Admin HTTP listener also serves bundled static Admin Web UI
assets at `/`, `/index.html`, `/styles.css`, and `/app.js`; those assets call
same-origin `/api/v1/*` and do not bypass the API contract.

Authenticated `GET /api/v1/trust-bundles`, CSRF-protected
`POST /api/v1/trust-bundles`, and CSRF-protected
`DELETE /api/v1/trust-bundles/{ref}` expose the managed private-PKI trust
bundle use cases. Import accepts `trust_bundle_ref` and `encoded_material`,
limits encoded material to 384 KiB, validates CA-only PEM through the injected
validator, and creates an immutable file-backed artifact under
`data/trust-bundles`. List and mutation responses contain only the logical ref,
certificate count, and import timestamp; PEM bytes, filesystem paths, subject
identities, and private material are never returned. Delete checks every retained
config revision and returns conflict while any listener or upstream references
the bundle. Import/delete do not mutate the active Core snapshot; activation
occurs only through the validated config apply path.

All mutation endpoints require:

- authenticated admin session
- CSRF token

Config/proxy-host mutations additionally require:

- validation before apply
- `ConfigLifecycle::{apply_with_core, rollback_with_core}` for revision commit,
  audit, and runtime command acknowledgement
- `CoreCommandClient` port for runtime changes behind the lifecycle use case

Certificate mutations additionally require:

- ACME input parsed from JSON
- certificate storage through the `CertificateStore` port
- `edge-proxy` config-file startup uses the file-backed `CertificateStore`
  rooted at `data/certs`
- private key/certificate PEM omitted from API responses
- `CoreCommandClient` acknowledgement for `InstallCertificate`

Trust-bundle mutations additionally require:

- validation and storage through injected application ports
- create-only refs; an existing ref returns HTTP 409
- retained-revision reference guard; a referenced ref returns HTTP 409
- CA-only material with at most 32 certificates and 384 KiB encoded input
- Product/audit events limited to ref, count, outcome, and stable error code

`GET /api/v1/status` keeps `current_revision_id` as a backward-compatible alias
for the desired repository revision and explicitly separates it from the running
active revision and resource policy:

```json
{
  "version_prefix": "/api/v1",
  "current_revision_id": "rev-desired",
  "desired_revision_id": "rev-desired",
  "active_revision_id": "rev-active",
  "restart_required": true,
  "activation_state": "pending_restart",
  "desired_resource_policy": {
    "max_connections": 100,
    "max_inflight_payload_bytes": 33554432
  },
  "active_resource_policy": {
    "max_connections": 1024,
    "max_inflight_payload_bytes": 134217728
  },
  "live_resource_status": {
    "revision_id": "rev-active",
    "generation": 12,
    "used_payload_bytes": 4096,
    "payload_limit_bytes": 134217728,
    "active_connections": 3,
    "pressure": "normal"
  },
  "routes": 1,
  "services": 1,
  "certificates": 0
}
```

`live_resource_status` is `null` until the data plane publishes its first latest-only
snapshot. Its revision is the running Core revision, its byte values are logical
payload-ledger usage and limit rather than process RSS, and `pressure` is one of
`normal`, `pressured`, `exhausted`, or `failed_closed`. It never includes PID,
connection identity, owner charge, request data, path, or secret material.

`activation_state` is `aligned` or `pending_restart`. A restart-required apply
updates only desired fields. A successful hot apply advances both snapshots. A
rollback to active-equivalent restart-sensitive settings returns the status to
`aligned`. The endpoint does not read environment variables or lock/read the Core
resource ledger directly. The Core publishes a deduplicated immutable aggregate
through a nonblocking port; the Admin adapter reads only the latest mirror.

## Metrics Export

The optional Prometheus listener is a separate loopback-only adapter configured
through `[metrics]`. The data plane publishes nonblocking typed observations and
never depends on the listener, encoder, Admin API, or a retention store.

Authenticated operators can read the same immutable snapshot through exact
`GET /api/v1/metrics`. Query parameters and non-GET methods are rejected. The
JSON object contains `ready`, desired/applied generation, estimated encoded
bytes, bounded drop counts, and `counters`, `gauges`, and `histograms` arrays.
Each array is capped at 500 deterministic series. This endpoint is read-only and
does not enable Prometheus exposition or mutate runtime state.
Remote metrics exposure and retention remain unsupported in this phase.

## Durable Audit Query

`GET /api/v1/audit` is read-only and is not itself audited. Records are newest
first. Applying a filter starts a new cursor chain; clients must reuse only the
server-issued cursor for the same ledger generation. Unknown, duplicate,
malformed, oversized or stale cursor/query values return a stable 400 error.

The response contains `records`, `next_cursor`, `head` and `admission_state`.
Each record exposes only sequence, timestamp, kind, action, outcome, actor,
operation/request IDs, target kind/ID, bounded revision IDs and stable error
code. The Admin UI renders these fields with DOM text nodes and uses no browser
persistent storage. `degraded` or `failed_closed` admission does not hide
read-only history from an authenticated operator.

## Proxy Host Request

New clients use `upstreams[]`. Each `id` is a stable identity and must be unique
within the proxy host. The `health_check` object is optional; absence disables
active probes. `upstream_url` remains in responses and is accepted as a legacy
single-upstream input, but must not be used together with assumptions that only
one target exists.

```json
{
  "id": "app",
  "name": "App",
  "domains": ["app.example.com"],
  "path_prefix": "/",
  "upstreams": [
    {"id": "app-a", "url": "http://127.0.0.1:3000"},
    {"id": "app-b", "url": "http://127.0.0.1:3001"}
  ],
  "health_check": {
    "enabled": true,
    "path": "/ready",
    "interval_ms": 10000,
    "timeout_ms": 2000,
    "healthy_threshold": 2,
    "unhealthy_threshold": 3,
    "status_min": 200,
    "status_max": 399
  },
  "https_enabled": false,
  "letsencrypt_enabled": false,
  "redirect_http_to_https": false,
  "enabled": true
}
```

Admin API handlers must not mutate core state or revision pointers directly. They
create application-level snapshots, then call `ConfigLifecycle::apply_with_core`
or `ConfigLifecycle::rollback_with_core`. The lifecycle sends commands through
the `CoreCommandClient` trait in `edge-ports`, commits `current` only after
accepted command acknowledgement, and records audit events through `AuditSink`.
Other control-plane adapters must reuse the same boundary instead of depending
on `edge-admin-api` internals. The bound TCP listener must not commit a config
revision until the running data plane has acknowledged the apply command.

Proxy host read endpoints reconstruct the response from generated route/service
pairs in the canonical `ConfigSnapshot`. Until the config model stores proxy-host
metadata directly, `name` is derived from `id`; clients must treat `id` as the
stable editable identity.

## Upstream Health Status

`GET /api/v1/upstream-health` requires an authenticated session and is
read-only. Its response is operational state, not canonical configuration and
must never be written into a config revision:

```json
{
  "revision_id": "revision-42",
  "generation": 7,
  "upstreams": [
    {"service_id": "app", "upstream_id": "app-a", "status": "healthy"},
    {"service_id": "app", "upstream_id": "app-b", "status": "unhealthy"}
  ]
}
```

Items are sorted by service and upstream identity. `status` is one of
`disabled`, `unknown`, `healthy`, or `unhealthy`. The response intentionally
omits URL, health path, raw failure reason, headers, bodies, and credentials.
If the runtime health supervisor is unavailable, the endpoint returns the
stable typed error `RUNTIME_HEALTH_UNAVAILABLE` instead of stale state.

## Certificate Issue and Renew Requests

Issue stores the certificate issue adapter result under the `{id}` path segment
as the target `CertificateRef`, regardless of the adapter's returned internal
reference. MVP smoke uses the fake adapter; real Let's Encrypt is deferred.

```json
{
  "domains": ["app.example.com"],
  "account_email": "admin@example.com",
  "production": false,
  "terms_accepted": false
}
```

Renew reuses the existing stored certificate domains for `{id}` and accepts only
the ACME account/runtime decision fields:

```json
{
  "account_email": "admin@example.com",
  "production": false,
  "terms_accepted": false
}
```

Successful issue/renew responses include:

```text
request_id
certificate_ref
domains[]
source
not_after_epoch_seconds
commands_sent
```

The response never includes private key material or certificate PEM.
On Unix platforms the bound `edge-proxy` file store writes
`data/certs/{certificate_ref}/privkey.pem` with mode `0600`.

## Auth Runtime

For the current MVP slice, the admin password hash is loaded once at process
startup from `SecretStore` name `admin-password-hash`, backed by:

```text
<data_dir>/secrets/admin-password-hash.secret
```

If this secret is absent, the Admin API enters setup-required mode. In that
mode, `POST /api/v1/setup` accepts the first password hash, writes the secret,
and enables login without reading environment values during runtime. The setup
endpoint returns `ADMIN_SETUP_ALREADY_COMPLETE` after the secret exists.

Setup request body:

```json
{
  "password_hash": "hash"
}
```

Login request body:

```json
{
  "password_hash": "hash"
}
```

Successful login returns `csrf_token` in JSON and sets:

```text
sponzey_session=<id>; Path=/; HttpOnly; Secure; SameSite=Strict
```

## Manual Certificate Import

`POST /api/v1/certificates/{id}/import` is the MVP certificate mutation path.
It requires a session and `X-CSRF-Token`. JSON fields are `domains`,
`fullchain_pem`, `private_key_pem`, and optional
`expected_not_after_epoch_seconds`.

The adapter calls the application use case through
`CertificateMaterialValidator`, `CertificateStore`, `AuditSink`, and
`CoreCommandClient`. States are `Received`, `Validated`, `Stored`,
`InstallCommandSent`, then `Installed` or `Failed`. Command rejection restores
the previous store value or removes a new ref and preserves active TLS.
Manual import also verifies that every declared domain is covered by the
validated leaf certificate DNS identity before store, audit, or core command
mutation. The supported identity policy is exact DNS and one-label wildcard
DNS; IP SAN, URI SAN, email SAN, and CN fallback are rejected. Responses mask
the key. Product events are
`certificate.import.success|failure`; PEM is never returned or logged.
